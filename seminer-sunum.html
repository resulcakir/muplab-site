01Introduction

Why a "living" meta-analysis?

A conventional meta-analysis starts to age on the day it is published: once a new study appears in the field, the result no longer shows the full picture. A living meta-analysis solves this problem by setting up the analysis not as a one-time publication but as a continuously updated, versioned system.

In our setup, the protocol and analysis plan are locked before the analysis; the data are open on GitHub; the model code is in R with brms · bayesmeta. When a new study is published, the automated PubMed search reports it; if the study meets the eligibility criteria, the data are extracted, the model is rerun, and the site is updated with a new version.

The Bayesian setup fits this loop particularly well: each new study is an update that carries the current knowledge from prior to posterior — that is precisely the statistical meaning of the word "living". You can try this idea yourself below.

The tool

The tool that scans PubMed for us

What runs the second stop of the living loop is a small scanner the laboratory wrote itself. Every month it runs on its own, asks PubMed the protocol’s question, and brings us only the records that are genuinely new. Below is an animation of one scan run.

How it works

  1. A scheduled task starts the scanner automatically every month — no one has to remember.
  2. The query is kept deliberately broad: only the intervention terms (taVNS · tVNS · nVNS and their long forms); no learning/memory filter — the point is not to miss any candidate.
  3. The scan is incremental: only records added since the last successful scan are pulled; if there is an interruption, it resumes from the date it left off.
  4. New PMIDs are compared against the seen list; duplicates are skipped silently.
  5. Candidates are written as RIS + CSV (ready for screening), and an email alert goes to the team.

Narrowing is in human hands: the initial (v1.0) screening was done in Rayyan while the manuscript was being written; in the living updates each monthly batch is checked by the team directly against the protocol criteria. The tool catches; we decide.

Method

Feel the update: how the distribution changes as evidence is added

The core of Bayesian inference is one sentence: prior belief + data = posterior belief. In the panel below, add the registered meta-analysis’s 22 primary accuracy (ACC) effects one by one in publication order (2018 → 2026); watch how the posterior distribution narrows and moves away from zero with each study.

A narrower prior = a more skeptical starting point. As data accumulate, the choice of prior affects the result less and less — try it yourself.

  1. The prior is our starting skepticism. We begin the analysis with μ ~ N(0, 0.5): “the effect is probably small, and its direction is not yet certain.”
  2. Each study gets a say in proportion to its precision. Weight = 1/se²: a study with a narrow margin of error pulls the distribution more strongly.
  3. In a living meta-analysis this update is formally repeated. When a new study is published, the model is re-run in R with MCMC and the site is refreshed through versioning.

Speed–accuracy trade-off: does the gain come at a cost?

No speed–accuracy trade-off. A registered joint (bivariate) Bayesian model over the 11 matched studies reporting both accuracy and RT tests whether taVNS buys its accuracy gain at a speed cost. It does not: accuracy rises (g = +0.29 [0.17, 0.41]) with no cost to response time (RT g = +0.14 [−0.04, 0.36]); the two outcomes move together only weakly and uncertainly (ρ = 0.14 [−0.73, 0.86]), and the most probable pattern is dual benefit (93%). These averages come from the matched subset, which is why they differ from the full-corpus pools (ACC g = 0.15, RT g = 0.18).

07Infrastructure

How does this site work?

muplab.com deliberately runs on a minimal infrastructure: a static site with no server of its own. This is also the key to the sustainability of the living system — maintenance cost is low, every version is traceable, and the whole record is transparent.

Static publishing

The pages are plain HTML+CSS+JS, hosted on GitHub Pages. Every update is a git commit: who changed what, and when — all of it is traceable.

The right job in the right layer

The living-MA exploration panel computes its statistics in the visitor's browser (speed + privacy). Dynamic work such as forms and team data is handled by an API running on the laboratory's own server.

Versioned analysis

The registered R outputs (brms) are published as locked figures; each new round of evidence is a new version (v1.0 → v1.1…). The exploration panel and the registered layer never mix.

How was the Bayesian meta-analysis produced — and where is everything?

  1. Open data. Effect sizes (Hedges g + SE, domain labels, direction-standardized) live as CSV in the open repository: github.com/resulcakir/SR-BMA.
  2. Model in R. A random-effects normal–normal model in brms, estimated by Stan MCMC on the laboratory computer; priors μ ~ N(0, 0.5) and τ ~ Half-Normal(0.3); domain subgroups in the same model. All R scripts are in the same repository.
  3. Cross-checks. The pool is re-estimated with bayesmeta, and the classical backbone is metafor’s cluster-robust (RVE) model.
  4. Locked outputs. Posterior summaries and figures are exported from R as fixed PNGs, committed to this site as a git commit, and tagged as a version (v1.0) — see the live living-MA page.
